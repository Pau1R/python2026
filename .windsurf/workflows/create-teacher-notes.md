## 3. Структура файла notes.md

Файл должен быть лаконичным и кратким.
Без лишнего текста. только опорные формулировки для преподавателя.

### 3.1 Заголовок

- `# Урок <номер>. <Название>`

Краткое уточнение под заголовком при необходимости:
- аудитория: "Студенты без опыта программирования"

### 3.2 Цель урока

- Заголовок: `## Цель урока`
- Содержание:
  - список 1-3 коротких пунктов
  - формулировки простым языком
  - ориентированы на студента без опыта программирования

Пример типа формулировок:
- "Познакомить студентов с базовыми понятиями ..."
- "Научить выполнять простейшие операции ..."

### 3.3 Тезисы

- Заголовок: `## Тезисы`
- Формат:
  - иерархический список
  - обычно не более 5 элементов на каждом уровне вложенности
  - в исключительных случаях допускается больше. если это нужно для полноты темы
  - каждый тезис короткий и понятный
  - избегать избыточно длинных перечислений без необходимости
- Уровень объяснения:
  - не использовать термины без пояснения
  - каждое новое понятие кратко расшифровывать для человека без опыта

Дополнительно в тезисах явно отрази граничные технические моменты:
  - какие пограничные случаи нужно проговорить в рамках темы
  - где тема опирается на знания с предыдущего урока
  - где важно аккуратно сформулировать материал.
    чтобы не создать путаницу перед следующей темой

## 4. План проведения

- Заголовок: `## План проведения`
- Структура: 3 или 4 блока занятия
- Каждый блок обязательно включает:
  - часть теории на 15 минут
  - часть практики на 15-20 минут

### 4.1 Структура блока

Для каждого блока создай подзаголовок и однотипную структуру.

Общая схема блока:

- Название блока
- Цель блока (2-4 коротких пункта)

**Часть 1. Теория**

- время: 15 минут
- формат работы:
  - объяснение
  - при необходимости короткое демо
- ключевые тезисы теории для этого блока
  - обычно до 5 пунктов. можно больше в исключительных случаях

**Часть 2. Практика**

- время: 15-20 минут (подбирать по сложности темы)
- формат работы:
  - только индивидуальная практика
  - краткий разбор решений
- при необходимости кратко указать в notes.md:
  - какие файлы или репозиторий он должен обновить

Ограничения по структуре:

- общее количество блоков в плане: 3 или 4
- в любых списках по умолчанию стараться держаться в пределах 5 пунктов
- превышать это значение только там. где это необходимо для понятности темы

## 5. Описание практических заданий

Практика должна быть спроектирована так. чтобы по ней можно было автоматически
генерировать индивидуальные задания и проверять результаты через GitHub.

1. Практические задания только индивидуальные:
   - не планируй задания в парах или группах
   - все формулировки так. чтобы студент выполнял работу сам

2. Требования к результату задания:
   - студент должен подготовить файл или набор файлов с кодом или другим содержанием
   - результат должен загружаться в GitHub
   - в notes.md явно указать:
     - тип артефакта (скрипт, ноутбук, текстовый отчет и так далее)
     - пример имени файла или структуры папки

3. Для каждой практической части блока в плане при необходимости добавь краткую пометку:
   - какие файлы или репозиторий он должен обновить

4. Уровень сложности:
   - ориентируйся на студентов без опыта программирования
   - дроби большие задачи на несколько понятных подшагов
   - избегай сложных формулировок
   - используй знакомые повседневные примеры

## 6. Требования к стилю и проверке

1. Стиль:
   - короткие фразы
   - технический и понятный язык
   - минимум терминов без пояснений
   - избегать больших сплошных абзацев

2. Структура:
   - обязательно наличие разделов:
     - "Цель урока"
     - "Тезисы"
     - "План проведения"
   - для всех списков:
     - по умолчанию не более 5 пунктов
     - в исключительных случаях допускается больше. если это оправдано задачами урока

3. Проверка:
   - убедись. что:
     - папка урока и файл notes.md созданы при необходимости
     - количество блоков в плане 3 или 4
     - в каждом блоке явно выделены части "Теория" и "Практика" с нужными таймингами
     - для каждой практической части есть указание.
       что результат студент загружает в GitHub в виде файлов
